## Executive Summary

**린클(Lincle)**은 리니지 클래식 유저를 위한 **게임 특화 C2C 거래 중개 플랫폼**이다.

**문제**: 연간 500~1,500억원 규모의 리니지 클래식 아이템 거래가 네이버 카페, 디스코드, 오픈카톡에 흩어져 있다. 검색이 어렵고, 거래 상태 관리가 없으며, 사기 위험이 높다.

**솔루션**: 서버/아이템/강화수치를 구조화한 매물 등록 → 1:1 채팅 → 예약 → 거래 완료까지 한 모바일 앱에서 끝나는 거래 플로우. 후기/신뢰 시스템으로 안전한 거래 환경을 제공한다.

**핵심 차별화**: ①게임 특화 구조화 데이터 ②거래 실행 플로우 통합 ③모바일 우선 ④신뢰 시스템 ⑤무료 (기존 중개 플랫폼은 5~10% 수수료)

**시장**: TAM 1~2조원(한국 게임 아이템) → SAM 500~1,500억원(리니지 클래식) → SOM 25~150억원(3년 내 5~10%)

**수익 모델**: 무료 출시 → 프리미엄 매물(3개월) → 안전 거래 에스크로 수수료(6개월) → 프로 구독(12개월). BEP: DAU 5,000 기준 8개월.

**실행 계획**: 12주 MVP 개발 (NestJS + React Native + PostgreSQL). 1인 개발자 + AI 도구로 가능한 설계. 초기 burn rate 42~67만원/월.

**핵심 리스크**: 게임사 정책 변동, Cold Start 문제, 에스크로 없이 사기 방지. 각 리스크별 완화 전략 수립 완료.

**성공 기준**: 출시 30일 내 DAU 100, 첫 거래 완료 50건, 채팅 전환율 10%+.

## 1. 시장 분석 및 경쟁 환경

### 1.1 시장 규모와 기회

**리니지 클래식 거래 시장의 현실**

리니지 클래식은 한국 MMORPG 시장에서 20년 이상 지속된 게임으로, 핵심 유저층이 30~50대 남성으로 구성되어 있다. 이 유저층의 특징:

- 높은 구매력: 경제활동 인구 중심, 고가 아이템 거래에 거부감이 적음
- 강한 충성도: 수년~십수년 단위로 게임을 유지하는 장기 유저
- 거래 빈도: 아이템 강화/교체 주기에 따라 반복 거래 발생
- 거래 단가: 고가 장비 기준 수십만원~수백만원대 거래도 일상적

**시장 규모 추정 (보수적)**

- 리니지 클래식 MAU: 약 10~30만 (추정)
- 월간 거래 활동 유저 비율: 약 20~40%
- 월간 거래 건수 추정: 5~20만 건
- 건당 평균 거래액: 약 3~10만원 (소액~고가 혼합)
- 연간 거래 총액 추정: 약 180억~2,400억원 범위

→ **핵심 인사이트**: 시장 규모 자체는 대형 마켓플레이스 대비 작지만, 유저당 LTV가 매우 높고, 경쟁이 제대로 이루어지지 않는 "under-served" 시장이다.

### 1.1.1 TAM / SAM / SOM 프레임워크

**TAM (Total Addressable Market) — 한국 온라인 게임 아이템 거래 전체**
- 한국 게임 아이템 거래 시장 연간 약 1~2조원 규모 (추정, 아이템매니아+비공식 거래 포함)
- MMORPG 카테고리가 거래액 기준 약 40~50% 차지

**SAM (Serviceable Available Market) — 리니지 시리즈 거래**
- 리니지 클래식 + 리니지M + 리니지W 거래 시장: 연간 약 2,000~5,000억원 (추정)
- 리니지 클래식 단독: 연간 약 500~1,500억원

**SOM (Serviceable Obtainable Market) — 린클이 3년 내 확보 가능한 점유율**
- 리니지 클래식 거래의 5~10% 점유 목표: 연간 25~150억원
- 프리미엄 매물 + 안전 거래 수수료 기준 플랫폼 수익: 연간 1~7.5억원

### 1.3 린클의 경쟁 우위 (핵심 차별화)

**1) 게임 특화 구조화 데이터**
- 서버/아이템명/강화수치/옵션/거래방식을 구조화된 필드로 관리
- 일반 마켓플레이스의 "자유 설명" 방식 대비 검색/필터 품질이 압도적으로 높음
- 카테고리별 속성 템플릿으로 등록 편의성과 탐색 정확도 동시 향상

**2) 거래 실행 플로우 통합**
- "게시판 + 메신저"가 아니라 "발견 → 채팅 → 예약 → 실행 → 완료"가 하나의 서비스 안에서 연결
- 거래 상태가 명시적으로 관리되어 "예약중인데 다른 사람도 연락 옴" 혼란 제거
- 예약 카드/시스템 메시지로 합의 내용이 기록에 남음

**3) 신뢰 시스템**
- 거래 횟수, 후기, 응답성, 노쇼 이력을 종합한 신뢰 배지
- 단순 별점이 아닌 "거래 가능성 신호" 중심 설계
- 사기 의심 신호 자동 탐지 + 운영 에스컬레이션

**4) 모바일 우선 UX**
- 한 손 조작으로 등록~완료까지 가능한 플로우
- 기존 경쟁자 대부분 PC 웹 중심 → 모바일 경험 공백

**5) 운영 투명성**
- 분쟁 시 플랫폼 내 기록(채팅, 예약, 상태 변경) 기반 판단
- 사용자에게 가시성 패널티/제재 사유를 설명하는 투명한 운영

### 1.4 경쟁 해자(Moat)와 네트워크 효과

**A. 직접 네트워크 효과 (Same-side)**
- 판매자가 많아지면 → 다른 판매자가 시세 참고/경쟁을 위해 진입
- 구매자가 많아지면 → 다른 구매자가 더 나은 선택지를 위해 진입
- *강도*: 중간 — 게임 거래 시장은 소수 활발 유저가 다수 거래를 만듦

**B. 교차 네트워크 효과 (Cross-side)**
- 매물이 많아지면 → 구매자 유입 증가
- 구매자 문의가 많아지면 → 판매자 등록 동기 증가
- *강도*: 강함 — 양면 마켓플레이스의 핵심 플라이휠

**C. 데이터 네트워크 효과**
- 거래 이력이 쌓이면 → 시세 데이터 품질 향상 → 가격 판단 보조
- 후기가 쌓이면 → 신뢰 시스템 정확도 향상 → 거래 전환율 상승
- 검색/필터 사용 패턴 → 추천/랭킹 최적화
- *강도*: 중간~강함 — 시간이 갈수록 후발 주자의 진입 장벽이 높아짐

**D. 전환 비용 (Switching Cost)**
- 낮은 초기 전환 비용: 기존 플랫폼에서 옮기기 쉬움 (우리에게 유리)
- 시간이 지나면 축적되는 전환 비용:
  - 거래 이력과 후기 → 다른 플랫폼으로 이동 시 신뢰도 리셋
  - 단골 거래 상대와의 관계 → 새 플랫폼에서 재구축 필요
  - 알림/찜/저장검색 설정 → 이전 불가

**E. 방어 전략**
1. **서버별 커뮤니티 밀착**: 각 서버 유저들의 거래 허브가 되면 교체 비용 급상승
2. **거래 이력 기반 신뢰**: 축적된 후기/이력은 경쟁자가 복제 불가
3. **운영 품질**: 분쟁 처리 경험과 정책 노하우가 쌓이면 품질 격차 발생
4. **데이터 자산**: 시세/거래 패턴 데이터는 시간이 갈수록 가치 증가

### 1.6 시장 진입 타이밍

**유리한 요인:**
- 리니지 클래식 리마스터 등 IP 리뉴얼 트렌드로 유저 재유입 가능성
- 기존 거래 플랫폼의 UX 정체 → 전환 비용이 낮음
- 고연령 유저의 스마트폰 숙련도 상승 → 모바일 앱 수용성 증가

**불리한 요인:**
- 게임사의 현금거래 정책 변동 리스크 (약관상 금지, 묵인 상태)
- 유저 기반이 고령화되어 장기적 시장 축소 가능성
- 규제 리스크 (게임 아이템 거래 관련 법적 논의)

## 2. MVP 기능 우선순위와 Cut-line

### 2.1 MVP 핵심 원칙

> "거래 성사율을 높이는 기능만 남긴다. 거래를 막는 기능은 최소로 유지한다."

MVP는 **첫 거래 완료 경험**을 최소 마찰로 만드는 것이 목표다. 복잡한 정책 엔진이나 고도화된 신뢰 시스템은 Post-MVP로 미룬다.

### 2.2 MVP 기능 Tier

**Tier 1: 거래 성사 필수 (반드시 포함)**
| 기능 | 이유 |
|------|------|
| 회원가입/로그인 | 사용자 식별 기본 |
| 매물 등록 (sell/buy) | 거래 대상 생성 |
| 매물 목록/상세 | 발견과 판단 |
| 서버/카테고리/가격 필터 | 탐색 효율 |
| 1:1 채팅 | 합의 도구 |
| 예약 제안/확정/취소 | 약속 구조화 |
| 매물 상태 변경 | 거래 진행 표현 |
| 거래 완료 처리 | 종결 |

**Tier 2: 신뢰/안전 최소치 (MVP 포함, 최소 구현)**
| 기능 | 이유 |
|------|------|
| 기본 후기 (추천/비추천 + 코멘트) | 신뢰 부트스트랩 |
| 신고 접수 | 악성 행위 대응 창구 |
| 관리자 신고 조회/처리 | 운영 가능성 확보 |
| 차단 기능 | 괴롭힘 방지 최소치 |
| 푸시 알림 (예약/메시지/상태 변경) | 거래 놓침 방지 |

**Tier 3: 탐색/재방문 촉진 (MVP 포함, 간략 구현)**
| 기능 | 이유 |
|------|------|
| 찜 (관심 목록) | 재방문 동기 |
| 알림함 | 푸시 미수신 시 보조 |
| 프로필 조회 | 상대 신뢰 판단 |
| 내 매물/내 거래 관리 | 판매자 운영 도구 |

**Tier 4: Post-MVP (출시 후 1~2개월 내 추가)**
| 기능 | 이유 |
|------|------|
| 가격 제안/역제안 구조화 | 협상 UX 고도화 |
| 저장 검색 + 알림 | 재방문 루프 |
| 매물 끌어올리기 | 노출 관리 |
| 신뢰 배지 세분화 | 신뢰 고도화 |
| 가시성 패널티 | 품질 관리 |
| 노쇼 판정 시스템 | 운영 고도화 |

**Tier 5: 장기 확장 (3개월+)**
| 기능 | 이유 |
|------|------|
| 에스크로/안전결제 | 사기 방지 고도화 |
| 사기 의심 자동 탐지 | 안전 고도화 |
| 아이템 카탈로그 표준화 | 데이터 품질 |
| 시세 추이 | 가격 투명성 |
| 반복거래/단골 | 리텐션 |

### 2.3 MVP Cut-line 결정 기준

**포함 기준:**
- 이 기능 없이 첫 거래가 완료될 수 없는가?
- 이 기능 없이 심각한 안전 사고가 발생하는가?
- 이 기능이 사용자 재방문에 직접 기여하는가?

**제외 기준:**
- "있으면 좋지만" 없어도 거래는 성사되는 기능
- 운영 효율 최적화 (MVP에서는 수동 운영 허용)
- 고도화된 정책 엔진 (placeholder로 대체)

### 3.1 아키텍처 결정 원칙

1. **MVP 속도 우선**: 4주 내 배포 가능한 스택
2. **모바일 우선**: 네이티브 앱 또는 하이브리드 + API 서버

### 3.3 핵심 아키텍처 결정

**A. 상태 머신 설계**
- 거래 상태(Listing status)와 예약 상태(Reservation status)를 분리
- 상태 전이는 서버 guard 함수에서 강제 (클라이언트는 허용 액션만 표시)
- 전이 이력은 별도 테이블(StatusHistory)에 append-only로 기록

**B. 채팅 아키텍처**
- 메시지 저장: PostgreSQL (정합성 우선, MVP 규모에서 충분)
- 실시간 전달: SSE (모바일 앱 → 서버 폴링 + SSE push)
- 읽음 커서: 사용자 단위 lastReadMessageId (Redis 캐시 + DB 영속)
- 시스템 메시지: 예약/상태 변경 이벤트를 chat timeline에 자동 삽입

**C. 멱등성과 동시성**
- 예약 확정: DB 레벨 advisory lock 또는 SELECT FOR UPDATE
- 매물 상태 변경: version 컬럼으로 optimistic locking
- 메시지 전송: 클라이언트 생성 idempotency key로 중복 방지

**D. 이벤트 발행**
- MVP: 동기적 in-process 이벤트 (EventEmitter 패턴)
- 이후: Outbox 패턴으로 비동기 전환 준비
- 이벤트 소비자: 알림 발송, 읽기 모델 갱신, 감사 로그

### 3.4 채팅 동기화 전략 상세

PRD에서 채팅을 "단순 메시지 리스트가 아니라 event timeline"으로 정의하고 있으므로, 동기화 설계가 핵심이다.

**MVP 동기화 아키텍처:**
```
[모바일 앱]                              [서버]
    |                                       |
    |-- POST /messages (메시지 전송) ------->|
    |                                       |-- DB 저장
    |                                       |-- SSE broadcast
    |<-- SSE event (새 메시지) -------------|
    |                                       |
    |-- GET /messages?after=cursor -------->|  (재연결 시 backfill)
    |<-- 누락 메시지 목록 ------------------|
```

**핵심 설계 결정:**

1. **커서 기반 동기화**: 각 참여자의 `lastReceivedEventId`를 기준으로 gap 감지 및 복구
2. **메시지 dedup**: 클라이언트가 생성한 `clientMessageId`로 중복 전송 방지
3. **읽음 커서 분리**: `lastReadMessageId`는 사용자 단위, `lastReceivedEventId`는 디바이스 단위
4. **SSE 연결 수명**: 모바일 백그라운드 진입 시 연결 해제, 포그라운드 복귀 시 재연결 + backfill
5. **시스템 메시지**: 예약/상태 변경 이벤트가 chat timeline에 자동 삽입 → 별도 messageType

**확장 경로 (Post-MVP):**
- SSE → WebSocket (양방향 필요 시)
- 단일 서버 → Redis Pub/Sub 기반 다중 인스턴스
- DB polling → Change Data Capture (CDC) 기반 실시간 projection

### 3.5 확장성 계획

**MVP 규모 (DAU 1,000 이하):**
- 단일 서버 인스턴스로 충분
- PostgreSQL 단일 노드
- SSE 동시 접속 ~200 정도 문제없음

**성장 단계 (DAU 5,000~10,000):**
- 서버 2~3대 수평 확장 (stateless API)
- Redis Pub/Sub로 SSE 이벤트 팬아웃
- PostgreSQL read replica 도입
- CDN 이미지 캐싱

**대규모 (DAU 50,000+):**
- 채팅 메시지 DB 분리 (별도 DB 또는 Cassandra)
- 이벤트 드리븐 아키텍처 (Kafka/NATS)
- 검색 전용 인덱스 (Elasticsearch/Meilisearch)

→ **원칙**: MVP에서 과도하게 설계하지 않되, 확장 시 재작업이 최소화되도록 인터페이스를 분리

**핵심 설계 결정:**
- Listing과 Reservation은 1:N이지만, 활성 예약은 매물당 최대 1개 (유니크 제약)

- Review는 TradeCompletion 1:2 (양측 각 1건)
- Report의 target은 다형성 (listing/user/chatroom/message/review)
- StatusHistory는 모든 상태 변경을 기록하는 이벤트 소싱 준비 테이블

### 3.7 보안 설계

**A. 인증/인가**
- JWT access token (15분 만료) + refresh token (30일, httpOnly cookie 또는 secure storage)
- 모든 API에 인증 미들웨어 (공개 매물 조회 제외)
- 역할 기반 접근 제어: user, moderator, admin
- 소유권 검증: 매물 수정/채팅 접근 시 본인 확인

**B. 데이터 보호**
- DB 비밀번호: bcrypt (cost factor 12)
- 민감 정보 로그 제외: 토큰, 개인정보 마스킹
- API 응답에 내부 ID 노출 최소화 (공개 slug 사용)
- 채팅 메시지 E2E 암호화는 MVP 범위 외 (서버 측 저장)

**C. API 보안**
- Rate limiting: IP + 유저 단위 (100 req/min 기본)
- CORS: 모바일 앱 origin만 허용
- Input validation: Zod/class-validator로 모든 입력 검증
- SQL injection: ORM 사용으로 기본 방어
- XSS: 채팅 메시지 sanitize

**D. 운영 보안**
- 관리자 채팅 열람 시 감사 로그 기록 필수
- 관리자 액션은 2FA 검토 (Post-MVP)
- 정기적 의존성 취약점 스캔 (npm audit)

### 3.8 대안 비교 및 기각 사유

| 대안 | 기각 사유 |
|------|----------|
| Flutter | React Native 대비 웹 개발자 풀이 작음, Expo OTA의 빠른 반복이 중요 |
| Go backend | 타입 공유 불가, 빠른 프로토타이핑에 불리, 소규모 팀에서 생산성 ↓ |
| MongoDB | 거래 정합성에 ACID 부족, 관계형 쿼리 많은 도메인에 부적합 |
| Firebase/Supabase | 복잡한 상태머신과 비즈니스 로직 제어에 한계, vendor lock-in |
| WebSocket | SSE 대비 인프라 복잡도 ↑, 양방향 불필요 (메시지 전송은 HTTP POST) |

| 대형 플랫폼의 게임 거래 진출 | 높음 | 낮 | 리니지 클래식 특화 깊이로 차별화, 커뮤니티 밀착 |

### 5.2 기술 리스크

|--------|------|------|----------|

### 5.3 운영 리스크

| 리스크 | 영향 | 확률 | 완화 전략 |
|--------|------|------|----------|

| 노쇼/분쟁 증가 | 중 | 중 | 노쇼 누적 기록, 신뢰도 반영, 예약 리마인드 강화 |

### 5.4 법적/규제 리스크 심층 분석

**A. 게임 아이템 현금 거래의 법적 지위**

현재 상황:
- 게임사 약관: 대부분 현금 거래 "금지" 명시 (리니지 클래식 포함)
- 현실: 수십년간 묵인, 아이템매니아 등 대형 플랫폼이 정상 영업 중
- 법적 판례: 게임 아이템은 "재산적 가치가 있는 디지털 자산"으로 인정되는 판례 존재
- 규제 동향: 게임산업법에서 직접 금지하지 않으나, "사행성 강화" 우려 시 규제 가능성

린클의 포지셔닝:
- "거래 중개 정보 제공 서비스"로 명확히 정의 (통신판매중개업 모델)
- 직접 결제/정산에 개입하지 않음 (MVP 단계)
- 에스크로 도입 시 전자금융업/PG 사업자 등록 검토 필요
- 약관에 "플랫폼은 거래 당사자 간 분쟁에 대해 직접적 책임을 지지 않음" 명시

**B. 필요한 사업자 등록/신고**

| 항목 | 필요 여부 | 시점 | 비고 |
|------|----------|------|------|
| 사업자 등록 | 필수 | 즉시 | 개인/법인 |
| 통신판매업 신고 | 필수 | 즉시 | 정보 중개 서비스 |
| 개인정보 처리방침 | 필수 | 출시 전 | 개인정보보호법 |
| 부가통신사업 신고 | 검토 필요 | 출시 전 | 앱 서비스 제공 |
| 전자금융업 등록 | 에스크로 도입 시 | Phase 3 | PG 연동 검토 |

**C. 개인정보보호법 대응**

- 수집 항목 최소화: 로그인 정보, 닉네임, 프로필 (실명/전화번호 미수집)
- 채팅 보관 기간: 거래 완료 후 90일 → 이후 자동 삭제 (분쟁 대비)
- 운영자 열람: 신고 처리 시에만, 감사 로그 기록 필수
- 위치 정보: 수집하지 않음 (거래 장소는 자유 텍스트)
- 개인정보 처리방침, 이용약관 작성 필요 → 출시 전 법률 자문 권장

**D. 면책 조항 핵심 포인트**

1. 플랫폼은 거래 대상 물품의 진위, 품질, 가격 적정성을 보증하지 않음
2. 거래 당사자 간 분쟁에 대한 직접적 법적 책임을 부담하지 않음
3. 게임사 약관 위반에 따른 게임 계정 불이익은 이용자 책임
4. 허위매물, 사기 행위 발견 시 계정 제한 및 관련 기록 보관 가능

## 6. 수익 모델과 사업 지속가능성

### 6.1 수익 모델 후보

**A. 광고 기반 (초기 무료 → 광고 수익)**
- 매물 목록/상세에 배너 광고
- 게임 관련 타겟 광고 (PC방, 게이밍 기어 등)
- 장점: 유저 진입 장벽 제로
- 단점: 트래픽 의존, UX 저해

**B. 프리미엄 매물 (끌어올리기/하이라이트)**
- 기본 등록은 무료, 상위 노출은 유료
- 1회 끌어올리기: 500~1,000원
- 하이라이트 배지: 1,000~2,000원/일
- 장점: 거래 성사율 직결, 유저 수용성 높음
- 단점: 초기 유저 적을 때 가치 부족

**C. 거래 수수료**
- 거래 완료 시 일정 비율 수수료 (1~3%)
- 에스크로 서비스와 연동 시 자연스러운 과금 포인트
- 장점: 거래 볼륨에 비례하는 안정적 수익
- 단점: 에스크로 없이는 수수료 강제 어려움, 우회 거래 유인

**D. 구독 (프로 판매자)**
- 월정액으로 등록 수 확대, 통계, 일괄 관리 도구 제공
- 월 9,900~29,900원
- 장점: 안정적 MRR
- 단점: 초기에 구독 가치 증명 어려움

**E. 안전 거래 수수료 (에스크로)**
- 플랫폼이 대금을 일시 보관, 양측 확인 후 정산
- 거래 건당 3~5% 또는 고정 수수료
- 장점: 사기 방지 + 자연스러운 과금, 가장 높은 단위 수익
- 단점: PG/에스크로 인프라 필요, 규제 검토 필요, MVP 범위 초과

### 6.2 권장 수익 전략 (단계별)

**Phase 1 (출시~3개월): 완전 무료**
- 목표: MAU 확보와 유저 습관 형성
- 수익: 없음 (투자금 또는 부트스트래핑)
- 핵심: "이 서비스가 없으면 불편하다"는 인식 심기

**Phase 2 (3~6개월): 프리미엄 매물 도입**
- 기본 등록 무료 유지
- 끌어올리기(500~1,000원) + 하이라이트(1,000~2,000원) 출시
- 예상 월 수익: DAU 1,000명 기준, 전환율 5% → 약 75~150만원/월

**Phase 3 (6~12개월): 안전 거래 + 구독 모델**
- 에스크로 기반 안전 거래 도입 (수수료 3~5%)
- 프로 판매자 구독 (월 14,900원)
- 예상 월 수익: 월간 거래 1,000건 × 5만원 × 3% + 구독 50명 → 약 225만원/월

**Phase 4 (12개월+): 플랫폼 확장**
- 다른 게임 서버 추가
- 광고 매체 도입 (게임 관련 타겟)
- API 파트너십 (시세 데이터, 커뮤니티 연동)

### 7.1 초기 유저 확보 (Pre-launch ~ 출시 1개월)

**A. 리니지 커뮤니티 침투**
- 주요 카페/디스코드/오픈카톡방에서 초기 테스터 모집
- "거래글 올릴 때 린클 링크도 같이 올려주세요" 캠페인
- 커뮤니티 주요 거래 유저에게 1:1 연락, 얼리 어답터 확보
- 인벤/루리웹 리니지 클래식 게시판 활용

**B. Cold Start 해결 전략 (양면 마켓 최대 난제)**

*문제*: 매물이 없으면 구매자가 안 오고, 구매자가 없으면 판매자가 등록 안 함.

*해결 전략 1: 서버별 집중 런칭 (Geo-focused Launch)*
- 전체 서버 동시 런칭이 아니라, 거래 활발한 2~3개 서버에 집중
- 해당 서버 커뮤니티에 밀착 → 해당 서버에서 "린클이 표준"이 되면 확산
- 당근마켓이 판교 → 강남 → 서울 확산한 것과 동일한 전략

*해결 전략 2: 초기 매물 시딩*
- 기존 카페/디스코드 거래글을 참고해 초기 매물 100~200개 직접 등록 (에디터 역할)
- 초기 판매자에게 인센티브: 끌어올리기 무료 쿠폰 10회, 프로필 "얼리 어답터" 배지
- "린클에 올리면 카페에도 링크 공유해드립니다" 번들 제안

*해결 전략 3: 공급자 우선 (Supply-first)*
- 판매자 확보가 구매자 확보보다 중요 (매물이 있어야 검색 트래픽 발생)
- 판매자에게 먼저 가치 제공: "조회수 통계, 찜 알림, 채팅 미리알림"
- 구매자는 자연 검색(ASO)으로 유입 → 매물이 있으면 자연스럽게 전환

**C. 레퍼럴 프로그램**
- "친구 초대 시 양쪽에 끌어올리기 무료 쿠폰"
- 초기에는 단순하게 시작, 이후 고도화

### 7.2 성장 채널 (출시 1~6개월)

**A. 검색 최적화 (ASO/SEO)**
- 앱스토어: "리니지 클래식 거래", "리니지 아이템 거래" 키워드 최적화

**B. 콘텐츠 마케팅**
- "리니지 클래식 시세 동향" 콘텐츠 주기적 발행
- "안전한 게임 거래 가이드" 콘텐츠
- 유튜브/스트리머 협업 (리니지 클래식 방송인)

**C. 커뮤니티 파트너십**
- 주요 길드/혈맹 단위 파트너십
- 서버별 거래 커뮤니티와 협업
- 게임 이벤트 시즌에 맞춘 프로모션

### 7.3 리텐션 전략

**A. 거래 완료 루프 강화**
- 거래 완료 → 후기 → 신뢰 배지 상승 → 더 많은 문의 → 재거래
- "거래 완료!" 축하 화면 + 공유 기능

**B. 재방문 트리거**
- 찜한 매물 가격/상태 변경 알림
- "이 서버에 새 매물이 올라왔어요" 저장 검색 알림
- 미완료 거래 리마인드

**C. 판매자 도구**
- 조회수/찜수/채팅수 대시보드
- 매물 관리 편의 기능
- 반복 판매자를 위한 빠른 재등록

### 8.1 페르소나

**페르소나 A: "전업 트레이더" 민수 (38세, 회사원)**

- 린클에 기대하는 것: 빠른 매물 등록, 진지한 구매자 필터링, 노쇼 방지

**페르소나 B: "가끔 사는" 현우 (42세, 자영업)**
- 리니지 클래식 3년차, 켄라우헬 서버
- 월 1~2회 구매, 강화 장비 위주
- 현재 인벤 게시판 + 지인 소개로 거래
- 불만: "원하는 템 찾기 어려움", "사기꾼인지 아닌지 모르겠음"
- 린클에 기대하는 것: 쉬운 검색, 판매자 신뢰도 확인, 안전한 거래

**페르소나 C: "운영자" 지은 (30세, 린클 운영팀)**
- 허위매물 신고 처리, 분쟁 중재
- 하루 50~100건 신고 큐 처리 목표
- 필요한 것: 빠른 증거 확인, 일괄 처리, 판단 기준 가이드

### 8.2 핵심 사용자 시나리오 (End-to-End)

**시나리오 1: 첫 판매 성공**
```
민수: 집행검 +9 판매하고 싶음
1. 앱 열기 → 매물 등록 (판매, 바츠, 무기, 집행검, +9, 800만원, 협상가능)
2. 이미지 촬영 → 첨부
3. 거래 가능 시간: "평일 저녁 8시 이후, 주말 종일"
4. 등록 완료 → 3분 소요
5. 30분 후 푸시: "새로운 채팅이 시작됐습니다"
6. 채팅 확인 → 현우가 가격 문의
7. 750만원에 합의
8. 예약 제안: "내일 토요일 14시, 인게임 바츠서버 기란마을"
9. 현우 예약 확정 → 매물 상태 "예약중"
10. 토요일 14시 → 인게임 만남 → 거래 완료
11. 민수: 거래 완료 처리
12. 현우: 후기 작성 "빠른 거래, 추천!"
13. 민수: 신뢰 배지 업데이트
```

**시나리오 2: 노쇼 발생 및 대응**
```
1. 예약 확정 상태 → 약속 시간 도래
2. 한쪽이 나타나지 않음
3. 나타난 쪽: 채팅에서 "안 오시나요?" 메시지 → 30분 대기
4. "거래 불발" 처리 → 노쇼 사유 선택
5. 노쇼 신고 → 상대 프로필에 누적
6. 매물 → "거래가능" 복귀, 다른 채팅 상대와 진행 가능
```

**시나리오 3: 사기 의심 신고**
```
1. 구매자: 거래 완료 후 받은 아이템이 설명과 다름
2. 신고 접수: "허위매물" + 스크린샷 첨부
3. 관리자: 채팅 로그 + 매물 설명 + 신고 증빙 확인
```

### 12.1 "시세 그래프" — 가격 투명성 무기

**아이디어**: 매물 등록/완료 데이터를 기반으로 아이템별 시세 추이 그래프 제공

- 린클 데이터로 유의미한 시세 추이 자동 생성
- "최저가 알림", "시세 하락 전 판매" → 거래 촉진 트리거

**구현**: Post-MVP (데이터 1,000건+), 난이도 중

- 이전 채팅 이력 참조 + 새 예약 바로 시작

**구현**: Phase 4, 난이도 낮
**영향**: 리텐션 핵심, LTV 극대화

### 12.4 "안전 거래 체크리스트" — 사기 예방 UX

## 14. PRD vs MVP 갭 분석

PRD v2.12는 매우 정교한 canonical vocabulary와 정책 체계를 정의하고 있다. MVP에서는 이 중 핵심만 구현하고 나머지는 placeholder로 남긴다.

### 14.1 PRD에서 MVP로 가져갈 핵심 계약

| PRD 개념 | MVP 구현 수준 | Post-MVP |
|----------|-------------|----------|
| Listing 상태머신 (available/reserved/pending_trade/completed/cancelled) | 완전 구현 | 유지 |
| Reservation 상태머신 | 완전 구현 | 재일정 추가 |
| ChatRoom 상태 | 기본 구현 (open/deal_completed) | 나머지 상태 추가 |
| 후기 eligibility | 기본 (거래 완료 + 7일 내) | 봉인/수정/양측 공개 정책 |
| 신고/모더레이션 | 기본 접수 + 관리자 처리 | 자동 분류, SLA, 이의제기 |
| 가시성 패널티 | 미구현 | Phase 3+ |
| 사기 의심 탐지 | 미구현 | Phase 4+ |
| 노쇼 판정 시스템 | 기본 신고만 | 구조화된 판정 시스템 |
| 거래 준비도 스냅샷 | 미구현 | Phase 3+ |
| 보호조치 이의제기 | 미구현 | Phase 4+ |
| 콘텐츠 검수 케이스 | 미구현 (수동 운영) | 자동화 |
| 운영 큐 taxonomy | 기본 신고 큐만 | 전체 큐 시스템 |
| 알림 인박스 | 기본 목록 | 액션 상태, 우선순위 |
| 도메인 이벤트/아웃박스 | 동기 이벤트 | 아웃박스 패턴 전환 |

### 14.2 PRD의 canonical vocabulary → MVP 단순화 전략

PRD v2.12는 약 100개 이상의 canonical vocabulary (xxxState, xxxType, xxxPolicy 등)를 정의한다.
MVP에서는 이를 아래 원칙으로 단순화한다:

**원칙 1: enum은 최소 집합만 구현하되, 확장 가능하게 설계**
- DB: string 컬럼 + application-level validation (enum 추가 시 migration 불필요)
- API: 응답에 enum 값 포함, 클라이언트가 unknown 값 graceful 처리

**원칙 2: snapshot/projection 객체는 inline 응답으로 대체**
- PRD의 `tradeTrustSnapshot`, `executionReadinessSnapshot` 등 → API 응답에 필요한 필드만 flat하게 포함
- Post-MVP에서 별도 read model로 분리

**원칙 3: 정책 엔진은 하드코딩 → 설정 파일 → 동적 규칙으로 점진 진화**
- MVP: transition guard 함수에 직접 코딩
- Phase 3: JSON 설정 파일로 규칙 외부화
- Phase 5+: 동적 정책 엔진

**원칙 4: 운영 vocabulary는 코드 상수 → 운영 DB로 점진 이동**
- MVP: action code, reason code를 TypeScript const로 관리
- Post-MVP: 운영자가 사유 코드를 추가/수정할 수 있는 관리 기능

### 14.3 PRD가 해결하지 못한 열린 질문

| # | 질문 | MVP 기본값 | 확정 필요 시점 |
|---|------|-----------|--------------|
| Q1 | currencyType을 어떻게 처리? (아데나/원화/혼합) | 자유 텍스트 + priceType enum | Phase 1 |
| Q2 | 인게임 거래 vs 오프라인 거래의 UX 차이 | tradeMethod 필드로 구분, UI 동일 | Phase 1 |
| Q3 | 에스크로 없이 사기 방지 어떻게? | 신고 + 후기 + 신뢰 배지 + 운영 대응 | 지속 |
| Q4 | 서버 마스터 데이터 유지보수 | 초기 수동 입력, 이후 크롤링/API 검토 | Phase 1 |

## 17. 피벗 시나리오

시장 반응에 따라 아래 피벗 경로를 고려한다.

### 12.1 피벗 A: 다른 리니지 시리즈 확장
- 트리거: 리니지 클래식 MAU 정체, 리니지M/W 유저 요청 증가
- 변경: 서버 마스터 데이터에 게임 구분 추가, 카테고리 확장
- 영향: 아키텍처 변경 최소, 운영 정책 추가 필요

### 12.2 피벗 B: 범용 게임 거래 플랫폼
- 트리거: 리니지 클래식 시장 축소, 다른 MMORPG 수요 확인
- 변경: 게임별 카테고리/속성 템플릿 시스템, 멀티 게임 지원
- 영향: 데이터 모델 확장 필요, 운영 복잡도 증가

### 12.3 피벗 C: 커뮤니티 + 거래 하이브리드
- 트리거: 거래보다 커뮤니티/정보 공유 수요가 더 큼
- 변경: 시세 게시판, 서버별 커뮤니티, 공략/정보 섹션 추가
- 영향: 콘텐츠 전략 필요, 수익 모델 변경 (광고 비중 ↑)

### 12.4 피벗 D: B2B 데이터 서비스
- 트리거: 거래 데이터의 시세/트렌드 분석 가치가 높음
- 변경: 시세 API, 리서치 리포트, 게임사/미디어 대상 B2B
- 영향: 데이터 파이프라인 구축 필요
