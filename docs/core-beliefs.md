# 핵심 원칙 (Core Beliefs)

이 문서는 에이전트가 애매한 상황에서 판단 기준으로 사용하는 팀의 엔지니어링 원칙이다.

## 코드 품질

- **단순함 우선** — 가장 단순한 해결책을 먼저 시도한다. 복잡한 추상화보다 명확한 코드
- **최소 영향** — 변경은 필요한 부분만 건드린다. 부수적 리팩토링은 별도로
- **임시 해결 금지** — 근본 원인을 찾는다. 우회(workaround)는 기술 부채

## 아키텍처

- **상태 전이는 Guard가 지킨다** — 비즈니스 로직에서 직접 상태를 변경하지 않는다
- **RBAC은 단일 매트릭스** — 권한 판단은 `RBAC_ACTION_MATRIX.md`가 유일한 원천
- **의존성은 안쪽으로** — domain ← guard ← handler 방향. 역방향 의존 금지

## 데이터

- **마이그레이션으로만 스키마 변경** — DDL 직접 실행 금지
- **시드 데이터는 코드로 관리** — 수동 INSERT 금지

## 운영

- **시크릿은 환경변수** — 코드, 커밋, 문서에 시크릿을 넣지 않는다
- **2GB RAM 제약** — NAS 배포 환경. 메모리 사용량을 항상 고려한다
- **Web 우선 배포** — Flutter Web으로 먼저 출시, 이후 모바일 확장
